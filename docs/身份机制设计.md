# 身份机制设计

## 设计原则

遵循 `Agent.md` 中的**权限延后**原则：认证框架先搭好，鉴权逻辑最后挂载。角色在用户表中定义为字段，业务逻辑按 `role` 字段判断。

## 角色定义

| 角色 | 值 | 来源 | 权限范围 |
|------|-----|------|----------|
| **管理员** | `admin` | 种子脚本创建 + 管理员手动提升 | 创建/删除知识库，修改用户角色，查看全量统计 |
| **普通用户** | `user` | 自主注册默认获得 | 上传文档，聊天，查看自己的统计 |

## 身份创建流程

```
                ┌────────────────────────┐
                │   系统启动               │
                │   seed.py 执行           │
                └───────────┬────────────┘
                            │ 检查是否存在 admin
                ┌───────────▼────────────┐
                │  admin 不存在？          │
                │  从 .env 读取初始管理员    │
                │  创建，role = "admin"    │
                └────────────────────────┘

                ┌────────────────────────┐
                │  普通用户注册             │
                │  POST /auth/register    │
                └───────────┬────────────┘
                            │ role 固定为 "user"
                ┌───────────▼────────────┐
                │  注册成功                │
                │  role = "user"          │
                └────────────────────────┘

                ┌────────────────────────┐
                │  管理员提升用户           │
                │  PUT /users/{id}/role   │
                │  需要 admin 权限         │
                └───────────┬────────────┘
                            │ 修改 role = "admin"
                ┌───────────▼────────────┐
                │  角色变更成功            │
                └────────────────────────┘
```

## 关键约束

1. **注册时：** 所有用户在 `UserCreate` 中不传递 `role` 字段，服务端硬编码为 `"user"`，防止前端伪造权限。
2. **种子脚本：** 仅在系统启动时运行一次，确保至少有一个管理员。
3. **环境变量：** 初始管理员的账号密码通过 `.env` 配置，Docker 部署时通过 `env_file` 注入，严禁写入镜像。
4. **Token 携带角色：** JWT Payload 中包含 `role` 字段，API 中间件读取后注入 `current_user`，无需每次查数据库。
