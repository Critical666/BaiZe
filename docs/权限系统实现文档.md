# 权限系统实现文档

> 本文档记录 BaiZe 权限体系的完整实现细节，基于 DEVELOPMENT_PLAN.md 中的方案A（admin/user 全局双角色）。

---

## 一、架构概览

```
用户请求 → Bearer Token → JWT 解码 → 获取 role 字段 → 权限判断
                                        │
                    ┌───────────────────┼───────────────────┐
                    ▼                   ▼                   ▼
              require_admin      get_current_user       公开接口
              (仅管理员可访问)    (登录即可访问)       (无需登录)
```

### 角色定义

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| `admin` | 管理员 | 全部功能：创建/删除知识库、查看全量统计、管理用户角色 |
| `user` | 普通用户 | 查看知识库、上传文档、聊天对话 |

## 二、后端实现

### 2.1 核心依赖（`backend/app/api/deps.py`）

新增 `require_admin` 依赖函数，基于已有的 `get_current_user` 实现：

```python
def require_admin(current_user: User = Depends(get_current_user)) -> User:
    """要求当前用户为管理员，否则返回 403。"""
    if current_user.role != "admin":
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="仅管理员可执行此操作")
    return current_user
```

**设计要点**：
- `require_admin` 内部依赖 `get_current_user`，形成依赖链：先验证 Token → 再验证身份 → 最后检查角色
- 未登录 → 401（由 `get_current_user` 抛出）
- 登录但非管理员 → 403（由 `require_admin` 抛出）

### 2.2 API 权限矩阵

| 路由 | 方法 | 权限要求 | 依赖函数 |
|------|------|----------|----------|
| `/api/v1/auth/register` | POST | 公开 | 无 |
| `/api/v1/auth/login` | POST | 公开 | 无 |
| `/api/v1/auth/me` | GET | admin | `require_admin` |
| `/api/v1/users/{id}/role` | PUT | admin | `require_admin` |
| `/api/v1/knowledge-bases` | POST | admin | `require_admin` |
| `/api/v1/knowledge-bases` | GET | 登录 | `get_current_user` |
| `/api/v1/knowledge-bases/{id}` | GET | 登录 | `get_current_user` |
| `/api/v1/knowledge-bases/{id}` | DELETE | admin | `require_admin` |
| `/api/v1/knowledge-bases/{id}/documents` | POST | 登录 | `get_current_user` |
| `/api/v1/knowledge-bases/{id}/documents` | GET | 登录 | `get_current_user` |
| `/api/v1/knowledge-bases/{id}/chat` | POST | 登录 | `get_current_user` |
| `/api/v1/stats` | GET | admin | `require_admin` |

### 2.3 新增接口

#### 获取当前用户信息
```
GET /api/v1/auth/me
Authorization: Bearer <token>
```

#### 修改用户角色
```
PUT /api/v1/users/{user_id}/role
Authorization: Bearer <admin_token>
Body: { "role": "admin" | "user" }
```

**安全约束**：
- 管理员不能修改自己的角色（防止最后一个管理员降权导致系统无管理员）
- 操作记录在日志中：`角色变更: xxx (user → admin), 操作者=yyy`

### 2.4 角色管理服务（`backend/app/services/auth_service.py`）

新增 `update_role` 方法：

```python
def update_role(self, user_id: str, new_role: str, operator_id: str) -> dict:
    if user_id == operator_id:
        raise ValueError("不能修改自己的角色")
    user = self.db.query(User).filter(User.id == user_id).first()
    if not user:
        raise ValueError(f"用户 {user_id} 不存在")
    old_role = user.role
    user.role = new_role
    self.db.commit()
    self.db.refresh(user)
    logger.info("角色变更: %s (%s → %s), 操作者=%s", user.email, old_role, new_role, operator_id)
```

## 三、前端实现

### 3.1 认证状态管理（`frontend/src/hooks/useAuth.ts`）

```typescript
export function useAuth() {
  // 从 localStorage 读取用户信息，提供 user / isAdmin / isLoggedIn
  // setAuth: 登录/注册成功后调用，存储 token + user 信息
  // clearAuth: 退出登录，清除 localStorage 并跳转 /login
}
```

**存储结构**：
- `localStorage.token` → JWT Bearer Token
- `localStorage.user` → JSON 序列化的 UserInfo 对象

### 3.2 请求拦截器（`frontend/src/api/client.ts`）

- 请求拦截器：自动携带 `Authorization: Bearer <token>`
- 响应拦截器：401 时自动清除 token 并跳转 `/login`（防止前端出现无限 401 循环）

### 3.3 路由守卫（`frontend/src/router.tsx`）

使用 `RequireAuth` 组件包裹所有需要登录的路由：

```typescript
function RequireAuth() {
  const token = localStorage.getItem('token');
  if (!token) {
    return <Navigate to="/login" replace />;
  }
  return <Outlet />;
}
```

未登录用户访问任意业务页面会被重定向到 `/login`。

### 3.4 UI 权限感知

| 页面/组件 | admin | user |
|-----------|-------|------|
| Layout 侧边栏 | 显示"统计面板"菜单项 | 隐藏"统计面板" |
| Layout 顶部栏 | 显示用户名 + "管理员"标签 | 显示用户名 + "普通用户"标签 |
| 首页 | 显示"新建知识库"按钮 + 删除按钮 | 隐藏创建/删除按钮，显示提示信息 |
| 首页 | — | 显示提示："你当前是普通用户，只能查看知识库和上传文档" |

## 四、初始管理员

系统启动时由 `AuthService.ensure_admin()` 自动创建，凭证从 `backend/app/core/config.py` 读取：

```python
init_admin_email: str = "admin@baize.com"
init_admin_password: str = "admin123"
init_admin_username: str = "admin"
```

**首次使用**：
1. 使用 `admin@baize.com` / `admin123` 登录
2. 进入"统计面板"确认系统状态
3. 通过 `PUT /api/v1/users/{id}/role` 提升其他用户为管理员

## 六、后续演进方向

当前方案A为最小可行权限体系，后续可按需扩展：

1. **小组/团队模型**：在 User 上叠加 Workspace/Group 层，实现多租户隔离
2. **细粒度权限**：从 admin/user 二元角色扩展为 RBAC（基于角色的访问控制）
3. **知识库级别授权**：admin 可将特定知识库的读写权限授予指定 user
4. **邀请机制**：admin 通过邮箱邀请 user 加入，生成邀请链接
