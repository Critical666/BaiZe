### Embedding 双模式设计

```
EMBEDDING_PROVIDER=local（默认）
  └─ sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
     输出维度: 384
     优点: 零配置、零成本、支持中文、离线可用
     缺点: 首次加载约 5-10 秒，模型体积约 420MB

EMBEDDING_PROVIDER=openai
  └─ text-embedding-3-small
     输出维度: 1536
     优点: 效果更好、无需本地模型
     缺点: 需要 API Key、有网络延迟和费用
```

> **维度联动**：`embedding_service.py` 导出 `EMBEDDING_DIM`，`vector_store.py` 直接引用，确保 Milvus Collection 维度与向量输出维度始终一致。

## 三、RAG 全流程（阶段 3 最终版）

### 文档处理流水线

```
用户上传文件
  ↓
DocumentService.upload()
  ├─ 写 Document 记录（status=pending）
  └─ _process_async()
       ├─ USE_CELERY=true  → process_document_task.delay()（Celery 异步）
       └─ USE_CELERY=false → _process_inline()（内联同步）
            ├─ parser.parse_document()     → PDF/Word/Markdown → 纯文本
            ├─ chunker.chunk_text()        → 500字+50重叠 → 文本块列表
            ├─ encode_texts(chunks)        → sentence-transformers 真实向量化
            │    └─ 全局单例 lazy 加载，normalize_embeddings=True
            ├─ vector_store.insert()       → Milvus Lite（384维，L2距离）
            └─ 更新 Document.status=done, chunk_count=N
```

### RAG 对话流程

```
用户提问
  ↓
ChatService.chat(kb_id, data, user_id)
  ├─ 1. encode_text(question)         → sentence-transformers 问题向量化
  ├─ 2. vector_store.search()         → Milvus Lite Top-K 检索（按 kb_id 隔离）
  ├─ 3. 构建 Prompt 上下文            → [来源: 文件名] + 文本块（每块截取 800 字）
  ├─ 4. llm_service.generate()       → OpenAI Chat Completion
  │    ├─ System: "你是白泽知识库助手..."（RAG 系统提示词）
  │    ├─ User: "参考文档内容：\n{context}\n\n用户问题：{question}"
  │    ├─ temperature=0.3, max_tokens=2048
  │    └─ 无 API Key 时回退 _fallback_answer()（关键词模板）
  ├─ 5. _save_history()               → ChatHistory 持久化（异步保存失败不影响主流程）
  └─ 返回 { answer, sources }
```

## 四、回退策略（阶段 3 最终版）

| 外部服务 | 不可用时的回退 | 触发条件 |
|----------|--------------|----------|
| Celery/Redis | 内联同步处理 `_process_inline()` | `USE_CELERY=false` 或 Celery 连接失败 |
| sentence-transformers | OpenAI Embedding API | `EMBEDDING_PROVIDER=openai` |
| OpenAI Embedding API | 本地 sentence-transformers | `EMBEDDING_PROVIDER=local`（默认） |
| OpenAI LLM API | 关键词提取 + 模板回答 | `OPENAI_API_KEY` 为空或 API 调用异常 |
| Milvus | Milvus Lite 嵌入式模式 | 默认行为，持久化到 `./milvus_data/baize.db` |

> 阶段 A2 中所有降级均为"替换"，阶段 3 实现了"平滑回退"——服务不可用时自动降级而非直接报错。

## 五、聊天记录持久化

### 新增数据表

```
chat_histories
├─ id          UUID PK
├─ kb_id       FK → knowledge_bases.id（索引）
├─ user_id     FK → users.id（索引）
├─ question    Text（用户问题）
├─ answer      Text（助手回答）
├─ sources     Text（JSON 数组，来源引用列表）
└─ created_at  DateTime（server_default=now()）
```

### 设计决策

1. **sources 存储**：使用 JSON 字符串存储而非关联表，因为来源仅用于展示，不需要反向查询。
2. **保存时机**：在 `ChatService._save_history()` 中用 try/except 包裹，保存失败仅记录日志不影响主流程。
3. **db 注入**：`ChatService` 构造函数新增 `db` 参数，通过 `get_chat_service` 依赖注入。无 db 时（如 Celery Worker）跳过保存。

### 统计联动

`StatsService.get_overview()` 中 `chat_count_7d` 从硬编码 0 改为：

```python
ChatHistoryService(self.db).count_recent(7)
```

查询 SQL 等价于 `SELECT COUNT(*) FROM chat_histories WHERE created_at >= now() - 7 days`。

## 六、Celery 接入设计

### 双模式切换

```python
# document_service.py
def _process_async(self, ...):
    if settings.use_celery:
        try:
            process_document_task.delay(...)
            return
        except Exception as e:
            logger.warning("Celery 提交失败，降级为内联处理: %s", e)
    self._process_inline(...)  # 默认路径
```

### 环境差异

| 环境 | USE_CELERY | 处理方式 | 说明 |
|------|-----------|---------|------|
| 本地开发 | `false` | 内联同步 | 无需 Redis，请求阻塞直到处理完成 |
| Docker 开发 | `false` | 内联同步 | 仅启动 Redis，后端本地热重载 |
| Docker 生产 | `true` | Celery 异步 | Worker 独立进程，大文件不阻塞 HTTP |

> 阶段 A2 中 Celery 任务使用 `ChatService._encode()` 生成向量，阶段 3 统一改为 `encode_texts()`，消除 ChatService 对 Embedding 的间接依赖。

# 后端（本地，热重载）

# 前端（本地，热重载）

- `/api/*` 反向代理到 `http://api:8000`

## 十、关键设计决策

### 1. EmbeddingService 独立为模块（而非嵌入 ChatService）

阶段 A2 中 `_encode()` 是 ChatService 的私有方法，导致 DocumentService 处理文档时需要 `from chat_service import ChatService; cs._encode()` 来调用 Embedding——违反封装原则。

阶段 3 将 Embedding 抽取为独立的 `embedding_service.py`，提供模块级函数 `encode_texts()` / `encode_text()`，ChatService 和 DocumentService 均通过公共接口调用，消除了跨模块的私有方法依赖。

### 2. LLMService 回退而非报错

`OPENAI_API_KEY` 为空或 API 调用异常时，`LLMService.generate()` 不抛异常，而是调用 `_fallback_answer()` 返回基于关键词匹配的模板回答。这确保了在未配置 LLM 时系统仍然可用——用户可以看到检索结果，只是回答质量较低。

### 3. ChatService.db 可选注入

`ChatService.__init__(self, db=None)` 中 db 参数可选。在 HTTP 请求链路中通过 FastAPI 依赖注入获得 db Session；在 Celery Worker 中如果需要调用 ChatService（目前不需要），可传入独立的 Session。这避免了 Celery Worker 中无法使用 FastAPI 依赖注入的问题。

### 4. EMBEDDING_DIM 单一来源

`embedding_service.py` 中根据 `EMBEDDING_PROVIDER` 动态计算维度（local=384, openai=1536），导出 `EMBEDDING_DIM` 常量。`vector_store.py` 通过 `from app.services.embedding_service import EMBEDDING_DIM` 引用，确保 Collection 维度与向量输出始终一致，避免维度不匹配导致 Milvus 写入失败。
