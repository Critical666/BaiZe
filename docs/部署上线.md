# 部署上线文档

> 本文档记录 BaiZe 从开发完成到正式上线（`wuyuxuan.xyz`）的完整部署过程，供后续运维和回滚参考。

| 条件 | 说明 |

## 二、部署方案

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| Docker Compose | 环境隔离、可复现 | PyTorch 镜像构建极慢（~1GB 依赖） |
| 手动部署 | 构建快、灵活 | 需手动管理进程 |

**最终选择：手动部署**。原因是 PyTorch + CUDA 依赖导致 Docker 镜像构建时间过长（532MB + 423MB），在当前服务器上手动部署更高效。

### 部署架构

```
用户浏览器 (HTTPS)
    │
┌───▼─────────────────────────────────────────┐
│  Nginx (SSL 终结 + 反向代理)                  │
│  - 443 → 静态文件 /var/www/rag-frontend       │
│  - /api/* → proxy_pass http://127.0.0.1:8000 │
└───┬─────────────────────┬───────────────────┘
    │                     │
┌───▼───┐          ┌──────▼──────┐
│静态文件│          │  FastAPI    │
│(构建产物)│          │  后端服务    │
└───────┘          └──────┬───────┘
                          │
                  ┌───────▼────────┐
                  │ SQLite + Milvus │
                  │   Lite         │
                  └────────────────┘
```

## 三、部署步骤

### 3.1 克隆代码

```bash
cd /root
git clone git@github.com:Critical666/BaiZe.git
cd BaiZe
```

### 3.2 后端配置与启动

```bash
cd /root/BaiZe/backend

# 创建环境变量文件（基于模板）
cp .env.example .env

# 编辑 .env，填入生产配置：
# - SECRET_KEY: 随机生成的强密钥
# - OPENAI_API_KEY: LLM API Key
# - CORS_ORIGINS: https://你的域名
# - INIT_ADMIN_*: 管理员初始凭证
vim .env

# 安装依赖
pip install -r requirements.txt

# 启动后端服务（后台运行）
nohup uvicorn app.main:app --host 127.0.0.1 --port 8000 > /root/nohup.out 2>&1 &
```

> 生产环境后端绑定 `127.0.0.1`，由 Nginx 反向代理暴露，不直接对外。

### 3.3 前端构建与部署

```bash
cd /root/BaiZe/frontend

# 安装依赖
npm ci

# 构建生产版本
npm run build

# 将构建产物复制到 Nginx 静态目录
mkdir -p /var/www/rag-frontend
cp -r dist/* /var/www/rag-frontend/
```

> 前端 API 请求使用相对路径（`baseURL` 为空），通过 Nginx 反向代理转发到后端。

### 3.4 Nginx 配置

创建 Nginx 站点配置：

```nginx
# /etc/nginx/sites-available/baize
server {
    listen 443 ssl;
    server_name wuyuxuan.xyz;

    ssl_certificate     /path/to/fullchain.pem;
    ssl_certificate_key /path/to/privkey.pem;

    # 前端静态文件
    location / {
        root /var/www/rag-frontend;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;
    }
}
```

```bash
# 启用站点
ln -s /etc/nginx/sites-available/baize /etc/nginx/sites-enabled/baize

# 检查配置并重载
nginx -t && systemctl reload nginx
```

### 3.5 验证

```bash
# 检查后端是否正常运行
curl http://127.0.0.1:8000/api/v1/auth/login

# 检查前端是否可访问
curl -I https://wuyuxuan.xyz
```

## 四、关键配置说明

### 4.1 前端 API baseURL

`frontend/src/api/client.ts` 中 `baseURL` 设为空字符串：

```typescript
baseURL: import.meta.env.VITE_API_BASE_URL || '',
```

这确保前端请求使用相对路径（如 `/api/v1/...`），由 Nginx 代理到后端。本地开发时可通过 `VITE_API_BASE_URL=http://localhost:8000` 指定。

### 4.2 CORS 配置

后端 `.env` 中 `CORS_ORIGINS` 必须包含生产域名：

```bash
CORS_ORIGINS=https://wuyuxuan.xyz
```

### 4.3 管理员密码

管理员账户仅在系统首次初始化时创建（数据库不存在时）。如果数据库已存在，修改 `.env` 中的 `INIT_ADMIN_*` 不会生效，需直接在数据库中修改。

## 五、Docker Compose 备选方案

如需使用 Docker Compose 部署，项目已提供完整配置：

```bash
# 后端环境变量
cp backend/.env.example backend/.env
vim backend/.env

# 一键启动（含 Redis、API、Worker、前端 Nginx）
docker compose up --build -d
```

**注意**：首次构建后端镜像需下载 PyTorch（CPU 版 ~532MB），构建时间较长。

### Docker 服务组成

| 服务 | 说明 | 端口 |
|------|------|------|
| redis | 缓存/消息队列 | 6379 |
| api | FastAPI 后端 | 8000 → 8008 |
| worker | Celery 异步文档处理 | - |
| frontend | Nginx 静态资源 | 80 → 8080 |

### 开发模式

```bash
docker compose -f docker-compose.dev.yml up --build -d
```

开发模式挂载源代码实现热重载，并额外暴露 3000 端口访问前端。

## 六、运维操作

### 更新部署

```bash
cd /root/BaiZe
git pull origin main

# 后端：重启服务
cd backend
pip install -r requirements.txt
pkill -f "uvicorn app.main:app"
nohup uvicorn app.main:app --host 127.0.0.1 --port 8000 >> /root/nohup.out 2>&1 &

# 前端：重新构建
cd ../frontend
npm ci && npm run build
cp -r dist/* /var/www/rag-frontend/
```

### 查看日志

```bash
# 后端日志
tail -f /root/nohup.out

# Nginx 日志
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log
```

## 七、已知问题与改进方向

| 问题 | 说明 | 改进方向 |
|------|------|----------|
| Docker 构建慢 | PyTorch 依赖过大 | 考虑预构建镜像或使用更轻量的 Embedding 方案 |
| 手动进程管理 | `nohup` 缺乏自动重启 | 可使用 `systemd` 服务管理后端进程 |
| 无 Celery Worker | 手动部署未启动异步文档处理 | 后续补充 Worker 或改用同步处理 |
| 管理员密码修改 | DB 已存在时 `.env` 不生效 | 提供密码重置命令或管理 API |
